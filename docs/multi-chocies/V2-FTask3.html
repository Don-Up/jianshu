Task 3: TipTap Rich Text Editor Integration

测试你对 TipTap 编辑器集成到 ArticleEditor 的理解程度。

为什么 TipTap 编辑器选择存储 HTML 而不是 Markdown 作为内容格式？
HTML 是 Web 原生格式，所有浏览器都原生支持渲染，无需转换层
Markdown 存储空间更小，可以节省数据库存储成本
前端框架偏好 HTML，Markdown 已被行业淘汰
关键洞察：TipTap 本身就是一个 HTML 编辑器，它的内部文档模型基于 DOM。调用 `editor.getHTML()` 直接获取 HTML 字符串，无需任何转换。如果使用 Markdown，TipTap 需要额外配置 markdown 解析/序列化扩展，增加了复杂性且没有实际收益。
// TipTap 直接输出 HTML，无需转换
const editor = useEditor({
  extensions,
  content,  // 这里可以直接传入 HTML 字符串
  onUpdate: ({ editor }) => {
    onChange(editor.getHTML());  // 直接获取 HTML
  },
});

// 如果想用 Markdown，需要额外的解析/序列化步骤
// import { generateHTML } from '@tiptap/html'
// const html = generateHTML(markdown, extensions);  // 多此一举
在 `extensions.ts` 中，为什么 StarterKit 的 `codeBlock` 需要配置为 `false`？
因为使用了 CodeBlockLowlight 替代默认的 codeBlock，以支持语法高亮
codeBlock 会导致安全漏洞，必须禁用
codeBlock 与 CodeBlockLowlight 功能重复，禁用可以提升性能
思维模型：TipTap 的 StarterKit 默认包含一个基础 codeBlock。CodeBlockLowlight 是增强版本，支持语法高亮。两者不能同时启用（会冲突），所以将 StarterKit 的 codeBlock 禁用，使用 CodeBlockLowlight 作为替代。这是一种"替换"而非"禁用"模式。
import StarterKit from '@tiptap/starter-kit';
import CodeBlockLowlight from '@tiptap/extension-code-block-lowlight';

export const extensions = [
  StarterKit.configure({
    codeBlock: false,  // 禁用默认的，使用增强版
  }),
  CodeBlockLowlight.configure({
    lowlight,
    HTMLAttributes: {
      class: 'bg-muted rounded-md p-4 font-mono text-sm',
    },
  }),
  // ...其他扩展
];
在 `TiptapToolbar` 中，点击"粗体"按钮会触发什么？
直接调用 `document.execCommand('bold')` 命令
调用 `editor.chain().focus().toggleBold().run()` 链式方法
设置编辑器的 `contentEditable` 状态为 true
TipTap 提供了链式 API（chain pattern）来执行命令。`chain()` 创建一个命令链，`focus()` 确保光标在编辑器内，`toggleBold()` 切换粗体状态，`run()` 执行整个链。这种设计允许组合多个命令（如 `chain().focus().toggleBold().toggleItalic().run()`），同时确保每个命令在正确的上下文中执行。
// TiptapToolbar 中的粗体按钮实现
{
  label: 'B',
  title: '粗体',
  action: () => editor
    .chain()
    .focus()
    .toggleBold()
    .run(),
  active: editor.isActive('bold'),
}

// 组合命令示例
editor
  .chain()
  .focus()
  .toggleBold()
  .toggleHeading({ level: 1 })
  .run();
为什么说 TipTap 是一个"headless"编辑器？
TipTap 只提供编辑逻辑和状态管理，不提供 UI，UI 完全由开发者自定义
TipTap 没有可视化的界面，只能在命令行中使用
TipTap 的代码没有 UI 相关的 CSS 类名
关键洞察：Headless 架构将"编辑能力"与"表现形式"分离。TipTap 负责创建编辑器实例、管理选区、处理输入事件，但它不知道也不关心按钮长什么样。开发者需要自己创建 Toolbar 组件，通过 `editor.chain()` 调用命令。这种模式给了开发者完全的 UI 控制权，同时也意味着更多的初始设置工作。
// TipTap 只提供核心能力
const editor = useEditor({
  extensions,
  content,
  onUpdate: ({ editor }) => { /* 状态变化回调 */ },
});

// EditorContent 是 TipTap 唯一提供的 UI 组件
// 它只渲染编辑器的 contenteditable 区域
return <EditorContent editor={editor} />;

// Toolbar 必须自己实现
// 它不是 TipTap 的一部分，而是你的应用组件
<div className="toolbar">
  <button onClick={() => editor.chain().focus().toggleBold().run()}>
    粗体
  </button>
</div>
当用户在 TipTap 编辑器中输入文本时，`onChange` 回调接收到的参数是什么？
用户输入的原始文本字符串
TipTap 编辑器生成的 HTML 字符串
包含选区信息的 JSON 对象
TipTap 的 `onUpdate` 回调在每次内容变化时被调用，回调参数是包含 editor 对象的事件。开发者通过 `editor.getHTML()` 获取当前内容的 HTML 表示。这个 HTML 是 TipTap 内部文档的序列化结果，包含了所有格式信息（如 ``、``、`

` 等标签）。

// TiptapEditor 中 onChange 的实现
const editor = useEditor({
  extensions,
  content,
  onUpdate: ({ editor }) => {
    // editor.getHTML() 返回 HTML 字符串
    const html = editor.getHTML();
    onChange(html);
  },
});

// 假设用户输入了 "Hello **World**"
// html 会是: "<p>Hello <strong>World</strong></p>"
在 `ArticleEditor` 中，为什么可以直接用 `TiptapEditor` 替换原来的 `Textarea`？
因为两者都暴露了 `content` 属性和 `onChange` 回调，接口兼容
因为 TipTap 和 Textarea 使用相同的 DOM 元素
因为 React 可以自动检测并替换 Textarea 为 TiptapEditor
关键设计决策：接口一致性使得替换成本最低。`Textarea` 用 `value` 和 `onChange`，`TiptapEditor` 用 `content` 和 `onChange`（稍有不同），但核心模式相同——都是"受控组件"，父组件通过回调接收内容更新。这种模式是 React 中组件组合的基础。
// Textarea 接口
<Textarea
  value={content}
  onChange={(e) => setContent(e.target.value)}
/>

// TiptapEditor 接口（类似）
<TiptapEditor
  content={content}
  onChange={setContent}
/>

// 替换后，ArticleEditor 的 state 逻辑完全不变
const handleSubmit = async (e) => {
  const articleData = {
    title,
    content,  // content 仍然是字符串，只是格式从 Markdown 变为 HTML
    // ...
  };
};
如果要在编辑器中插入一张图片，用户的操作流程是什么？
用户选择本地文件 → 上传到服务器 → 返回 URL → 插入编辑器
用户直接粘贴图片到编辑器 → TipTap 自动处理上传
用户在弹窗中输入图片 URL → 点击确认 → 通过 `setImage` 插入
当前实现使用 URL-only 方式：用户通过 `window.prompt()` 输入图片 URL，编辑器调用 `editor.chain().focus().setImage({ src: url }).run()` 插入。这是快速实现，但限制了用户只能使用网络图片。如果需要支持本地上传，需要添加文件上传 API 和 `FormData` 处理。
// TiptapToolbar 中的图片按钮实现
const handleImage = () => {
  // window.prompt 是简单的 URL 输入方式
  const url = window.prompt('输入图片 URL:');
  if (url) {
    editor
      .chain()
      .focus()
      .setImage({ src: url })
      .run();
  }
};

// 如果需要支持本地上传：
// 1. 创建 /api/upload 端点
// 2. 使用 FormData 上传文件
// 3. 将返回的 URL 传入 setImage
为什么 `useEditor` 必须在带有 `'use client'` 指令的组件中使用？
因为 TipTap 需要访问 `document` 对象来操作 DOM
因为 `useEditor` 是一个 React Hook，而 Hooks 只能在客户端组件中使用
因为 Next.js 要求所有使用第三方库的组件都必须标记为客户端组件
这是一个关于 React 基础的问题。`useEditor` 内部使用了 `useState` 和 `useEffect` 等 React Hooks，而 Hooks 的规则规定它们只能在组件函数顶层调用，不能在服务器组件中使用。`'use client'` 指令告诉 Next.js 这个模块是客户端代码，需要水合处理。
// 'use client' 是必须的，因为 useEditor 是 Hook
'use client';

import { useEditor, EditorContent } from '@tiptap/react';

export function TiptapEditor({ content, onChange }) {
  // useEditor 内部调用 useState, useEffect
  const editor = useEditor({...});

  return (
    <div>
      <EditorContent editor={editor} />
    </div>
  );
}
