ArticleEditor 组件如何区分「创建文章」和「编辑文章」两种模式？
通过 `isEditing` 属性判断 — 当 `isEditing=true` 时调用 `update` API，否则调用 `create` API
通过 `slug` 属性判断 — 如果传入了 `slug` 说明是编辑模式，否则是创建模式
通过 `initialData` 属性判断 — 如果有 `initialData` 说明是编辑现有文章
`isEditing` 是一个明确的布尔标志，直接表达开发者意图。代码中使用 `if (isEditing && slug)` 来决定调用哪个 API。这种方式比通过检查 `slug` 是否存在更清晰，因为编辑模式可能需要 `slug`，但 `slug` 的存在本身不一定代表编辑意图。
interface ArticleEditorProps {
  initialData?: Partial<CreateArticleRequest>;
  slug?: string;
  isEditing?: boolean;
}

let result;
if (isEditing && slug) {
  result = await articleApi.update(slug, articleData);
} else {
  result = await articleApi.create(articleData);
}
为什么标签 (tags) 在表单中使用字符串输入，但在 API 调用时需要转换为数组？
用户友好性 — 用户在表单中输入 "技术, 随笔, 读书" 比输入 `["技术", "随笔", "读书"]` 更自然
数据库存储 — 关系型数据库通常用 VARCHAR 存储逗号分隔的字符串比 JSON 数组更高效
API 兼容性 — CreateArticleRequest 定义 tags 为 `string[]` 类型，所以必须转换
这是一个用户体验 (UX) 设计决策。用户界面应该尽可能自然和简单 —— 人们习惯用逗号分隔词汇，而不是编程语言中的数组语法。后端 Prisma schema 可以用 `@db.Text` 存储 JSON 字符串，或用 `String[]` 支持原生数组。转换逻辑在前端处理，保持 API 接口类型安全。
// 表单状态
const [tags, setTags] = useState((initialData?.tags || []).join(', '));

// 提交时转换
tags: tags ? tags.split(',').map((t) => t.trim()).filter(Boolean) : undefined
为什么 `fetchApi` 函数的返回类型是 `Promise<ApiResponse<T>>` 而不是直接返回 `T`？
类型安全 — 强制所有 API 调用都必须处理 `success` 字段和可能的错误
一致性 — NestJS 的 TransformInterceptor 统一包装响应，前端必须匹配这种结构
性能 — 将响应包装在额外对象中可以启用 HTTP 缓存头
后端 NestJS 应用使用 `TransformInterceptor` 全局拦截器，将所有响应统一包装为 `{ success: boolean, data: T, ... }` 格式。前端 `fetchApi` 必须返回 `ApiResponse<T>` 以匹配后端实际返回的数据结构。如果直接返回 `T`，调用方无法访问 `success`、`error` 等元数据。
async function fetchApi<T>(endpoint: string, options?: RequestInit): Promise<ApiResponse<T>> {
  // ...
  return res.json(); // 后端返回 { success: true, data: {...} }
}

// 错误：直接返回 T
const articles = await articleApi.list(); // articles 是 ApiResponse，不是 Article[]

// 正确：访问 .data
const articles = res.data?.items || [];
表单提交失败时，为什么错误信息显示在 `catch` 块之外设置？
代码简洁性 — 减少 `catch` 块的代码量
区分错误来源 — API 返回的业务错误（如"标题重复"）vs 网络/服务器异常
符合 React 习惯 — setState 应该在 try-catch 外部调用以避免闭包问题
区分两种错误来源对用户体验至关重要：API 返回的业务错误如"标题已存在"、"未登录"有意义的用户提示；网络/服务器异常如 500 错误、连接超时需要通用错误信息。API 返回 `{ success: false, error: "标题已存在" }` 时走 `else` 分支，HTTP 状态码非 2xx 时走 `catch` 分支。两者都设置 `setIsSubmitting(false)` 在 `finally` 中确保按钮状态正确恢复。
try {
  let result;
  if (isEditing && slug) {
    result = await articleApi.update(slug, articleData);
  } else {
    result = await articleApi.create(articleData);
  }

  if (result.success && result.data) {
    router.push(`/article/${result.data.slug}`);
  } else {
    setError(result.error || 'Failed to save article'); // 这里也设置错误
  }
} catch (err) {
  setError(err instanceof Error ? err.message : 'Failed to save article');
} finally {
  setIsSubmitting(false);
}
用户点击「发布」按钮后成功，页面应该跳转到哪里？
跳转到文章详情页 — `/article/${result.data.slug}` — 让用户立即看到发布的文章
跳转到首页 — `/` — 让用户在文章列表中看到新发布的文章
停留在编辑页 — 保持当前页面，允许用户继续编辑
发布后跳转到文章详情页是最佳用户体验 (UX) 实践：即时反馈让用户立即看到自己创建的内容；成就感来自看到完成的文章；下一步操作可以继续阅读评论、分享链接。跳转到首页虽然能展示新文章，但用户可能不确定是否发布成功；留在编辑页则让用户困惑「到底发布成功没有」。
if (result.success && result.data) {
  router.push(`/article/${result.data.slug}`);
}
为什么 `PageLayout` 在 `write` 页面不需要 `showFooter={false}` 参数？
性能考虑 — 写文章页面需要所有 UI 元素，不应该省略 footer
页面长度 — 写文章页面内容足够长，footer 不会影响用户注意力
视觉设计 — PageLayout 的默认设计已经适合长页面布局
`showFooter={false}` 主要用于登录/注册等短页面，目的是减少视觉干扰让用户聚焦表单，避免页面内容过少时 footer 显得突兀。写文章页面有编辑器和大量表单内容，页面本身就很长，footer 自然在底部且不会干扰用户。这是自适应布局的设计 —— 短页面隐藏 footer，长页面显示 footer。
// 登录页
<PageLayout showFooter={false}>

// 写文章页
<PageLayout> // 没有 showFooter 参数，footer 会正常显示
如果用户快速连续点击「发布」按钮，会发生什么？如何防止？
正常处理 — 浏览器会自动 debounce 连续点击
发送多次请求 — 可能创建多篇重复文章
使用 `isSubmitting` 状态禁用按钮 — 防止重复提交
浏览器不会自动 debounce 表单提交。快速连续点击会触发多次 `handleSubmit` 调用，导致发送多个相同的 `POST /api/articles` 请求，后端可能创建多篇重复文章（除非有幂等性保护），用户体验混乱。解决方案是 `isSubmitting` 状态：`setIsSubmitting(true)` 在提交开始时设置，按钮通过 `disabled={isSubmitting}` 禁用，`finally` 块确保无论成功或失败，按钮状态都会恢复。
const [isSubmitting, setIsSubmitting] = useState(false);

// 表单提交开始时
setIsSubmitting(true);

// 按钮禁用
<Button type="submit" disabled={isSubmitting}>
  {isSubmitting ? '发布中...' : '发布'}
</Button>
在 `ArticleEditorProps` 中，为什么 `initialData` 使用 `Partial<CreateArticleRequest>` 类型？
部分更新 — 编辑时只需要传递用户修改的字段
可选属性 — 创建新文章时不需要传入 `initialData`
类型宽松 — 允许传入任何形状的数据而不做验证
`Partial<T>` 表示 T 的所有属性都变为可选，这在编辑场景中很有用：创建时不需要 `initialData`（`undefined`），编辑时传入现有文章数据，只填充用户修改的字段。关键点：`Partial` 不是为了"类型宽松"，而是为了表达「部分数据」的概念。
interface ArticleEditorProps {
  initialData?: Partial<CreateArticleRequest>;
  slug?: string;
  isEditing?: boolean;
}

// 创建模式
<ArticleEditor />

// 编辑模式
<ArticleEditor
  initialData={{ title: '原标题', content: '原内容', tags: ['tech'] }}
  slug="existing-slug"
  isEditing={true}
/>
