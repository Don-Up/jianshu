为什么用户主页使用 `Promise.all` 同时获取用户信息和文章列表，而不是顺序调用？
代码简洁性 — 减少嵌套层级，让代码更易读
性能优化 — 两个 API 请求并行执行，总耗时等于较慢的那个而非两者之和
错误处理 — Promise.all 可以一次性捕获所有错误
`Promise.all` 解决的是"请求瀑布"（request waterfall）问题。如果用顺序调用：`const profile = await userApi.getByUsername(); const articles = await userApi.getArticles();`，总耗时 = profile请求时间 + articles请求时间。使用 `Promise.all`：总耗时 = max(profile请求时间, articles请求时间)。对于需要同时显示用户信息和文章列表的页面，并行获取能显著减少用户看到白屏的时间。
// ❌ 顺序请求（总耗时 = 300ms + 200ms = 500ms）
const profile = await userApi.getByUsername(username);
const articles = await userApi.getArticles(username);
setProfile(profile);
setArticles(articles);

// ✅ 并行请求（总耗时 = max(300ms, 200ms) = 300ms）
const [profileRes, articlesRes] = await Promise.all([
  userApi.getByUsername(username),
  userApi.getArticles(username, { page: 1, limit: 20 }),
]);

数据来源 — isFollowing 是"我是否关注了这个用户"，属于本地 UI 状态而非全局认证状态

为什么 `useParams` 获取的 `username` 需要类型断言 `as string`？
安全性 — 防止 XSS 攻击，确保 username 是字符串而非恶意脚本
Next.js 的 `useParams` 返回 `Record` — 动态路由参数可能是单值或数组
TypeScript 默认将 URL 参数推断为 `string | undefined` — `as string` 提供非空断言
Next.js 的 `useParams` 返回类型是 `Record`，因为某些路由参数可能是数组（如 `/user/[...slug]`）。即使 `[username]` 明确是单值 segment，TypeScript 仍保守地推断为 `string | string[]`。使用 `as string` 告诉 TypeScript："我确定这个路由段是单值"，这是合理的因为 `/user/[username]` 中 `username` 肯定是字符串。如果不断言，访问 `params.username` 会得到 `string | string[]` 类型，无法直接用于 API 调用。
// useParams 返回类型
const params = useParams();
// params.username 的类型是 string | string[]

// 使用类型断言明确它是单值
const username = params.username as string;

// 然后才能直接使用
userApi.getByUsername(username); // ✅ username 是 string 类型
关注/取关操作失败时，为什么 UI 状态会回滚？
React 的 `useState` 在错误时自动重置之前的状态
错误被捕获后没有更新 `isFollowing` 状态，所以 UI 保持不变
乐观更新需要手动回滚 — 操作失败时显式恢复之前状态
这不是 React 自动行为，而是显式设计：`try { setIsFollowing(!isFollowing); /* API 调用 */ } catch { setIsFollowing(!isFollowing); /* 回滚 */ }`。乐观更新（optimistic update）的核心思想是"先更新 UI，立刻给用户反馈，再异步请求后端"。如果后端请求失败，必须回滚到之前状态。当前实现中，如果 `userApi.follow()` 抛出异常，`isFollowingLoading` 被设为 false 但 `isFollowing` 保持失败前状态（已取反），实际效果是非预期的。更好的做法是在 `catch` 中使用之前的状态值来回滚。
const handleFollow = async () => {
  const previousState = isFollowing; // 保存之前状态
  setIsFollowingLoading(true);
  try {
    setIsFollowing(!isFollowing); // 乐观更新
    const res = await userApi.follow(user.id);
    if (res.success && res.data) {
      setIsFollowing(res.data.isFollowing); // 确认状态
    } else {
      setIsFollowing(previousState); // 失败回滚
    }
  } catch (err) {
    setIsFollowing(previousState); // 异常回滚
  } finally {
    setIsFollowingLoading(false);
  }
};

用户页面是动态路由，骨架屏渲染时机难以确定

展示目的不同 — 标题行是"简短标签"（名字后面的小注释），侧边栏是"详细介绍"
占位需要 — 标题行需要填充内容以保持布局平衡
标题行的 bio 是"标签式展示"：在用户名后面用间隔号分隔，如果 bio 不为空则显示。这种模式常见于社交平台：`用户名 · 一句话简介`。侧边栏的 bio 是"详细介绍"，来自同一个字段但用途不同。这种重复是合理的 UI 设计，因为：1）用户快速扫过时能看到简短信息；2）想了解更多的用户可以看侧边栏的完整介绍。两者不是冲突的，而是满足不同阅读深度。

如果访问不存在的用户名（如 `/user/不存在`），页面如何处理？
显示 404 页面 — 后端返回 404，前端显示"用户不存在"
显示空页面 — 只有错误信息没有 404 状态码
重定向到首页 — 找不到用户就返回首页
当前实现中，如果 `profileRes.success` 为 false（API 返回 404 或其他错误），前端设置 `setError('User not found')`，然后在 error 状态渲染时显示错误信息。这是一个"软 404"处理——返回 200 但带错误信息，而非真正的 404 HTTP 状态码。对于 SEO 不利（搜索引擎不会识别这是 404 页面），但用户体验上能提供有意义的消息。如果需要真正的 404，应在 API 返回 404 时 throw error，让 Next.js 的 `notFound()` 处理。

为什么 `ArticleList` 组件接收 `articles` 作为 prop 而不是自己调用 API 获取？
性能考虑 — 避免重复 API 调用，数据已在父组件缓存
安全性 — 防止文章列表 API 被恶意爬取
关注点分离 — 列表组件只负责渲染，数据获取由页面层负责
这是一个"数据获取 vs 数据展示"分离的设计决策。如果 `ArticleList` 自己调用 API：1）每个用户主页都会多一次 API 调用；2）难以实现跨页面共享（如下拉加载更多）；3）测试时需要 mock 组件内部 API。当前设计中，父组件 `UserProfilePage` 负责获取数据，`ArticleList` 只接收已获取的 `articles` 数组并渲染。这种模式让组件更纯粹（pure）、更易测试、更容易共享数据获取逻辑。
